## Follow up from previous meetings/discussions

* (Sebastian): *very drafty branch:* https://github.com/numpy/numpy/compare/main...seberg:numpy:descr-custom-dtypes
  * I am thinking more about refining the `dtype.str` and `dtype.descr` mechanisms.  Nothing is perfect, honestly... but:
  * `dtype.str` should default to a new "name" (we want that I think), plus the byte-order, e.g. `>float64` (if float64 was a new dtype)
    * Caveat: parametric dtypes would return `|V<nbytes>` here, but we'll need a way to override this in the future (maybe via subclassing).  That would allow things like `<datetime64[ns]` then...
  * `dtype.descr` could be something like `("<float64", None)` where the `None` would be a dict of parameters, e.g. `("<datetime64", {"unit": "ns", "scaling_factor": 1})`...
     I.e. parametric types can return a dict and we then finalize via:
     `np.dtypes.Datetime64DType(unit="ns", scaling_factor=1, byte_order="<")` (byte-oder if needed).
  * Maybe: Just don't use `.descr`...
